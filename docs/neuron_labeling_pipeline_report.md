# Звіт: пайплайн «нейронного» лейблінгу (активні SAE-ознаки на текстах)

Цей документ описує, **як зараз** у репозиторії MamayScope збирається метадані про те, **для якого вхідного тексту** і **на якому токені** вважаються «активними» певні індекси після проходження активацій через навчений **sparse autoencoder (SAE)**. Формат зберігання — **Parquet** (шардовані файли `*_batch_*.parquet`, один **рядок** на вхідний текст, вкладена колонка `tokens`).

---

## 1. Важливе уточнення: що саме ми називаємо «нейронами»

У виводі пайплайна в полі `fired_features` фігурують **індекси ознак SAE** (розмірність словника ознак `d_sae`, наприклад 16 384), а **не** індекси «сирих» нейронів прихованого шару трансформера.

- **Чому так:** SAE навчений відтворювати активації в підпросторі; кожна координата в просторі ознак — це **feature** (часто кажуть «латент» або в інтерпретації — «нейрон SAE»), з яким в асоціації звикли працювати в mechanistic interpretability.
- **Що подаємо в SAE:** вектори активацій із **резидуального потоку** на цільовому шарі (див. нижче), узгоджені з тим, на чому навчався SAE (`hook_resid_post` для блоку `TARGET_LAYER` у конфігу тренування).

Якщо потрібен саме лейблінг **нативних** нейронів (розмірність `d_model`), це **інший** експеримент; поточний пайплайн спеціально заточений під **SAELens SAE**.

---

## 2. Архітектура процесу (загальна схема)

Пайплайн побудований як ланцюжок кроків `PipelineStep`, які послідовно викликає `PipelineExecutor` (той самий патерн, що й для генерації даних під тренування SAE).

```text
HFCsvBatchSource → MamayActivationProcessor → SaeFeatureEncodeProcessor → ParquetNeuronActivationSink
```

- **Точка входу:** `uv run -m src.pipelines.neuron_labeling_pipeline`
- **Конфігурація:** `src/core/settings.py` та змінні середовища / `.env` (`MODEL_ENDPOINT`, `TARGET_LAYER`, `BATCH_SIZE`, `NEURON_LABEL_*`, `SAE_*` тощо).

---

## 3. Покроковий розбір

### Крок A. Джерело текстів — `HFCsvBatchSource`

**Файл:** `src/pipelines/sources/hf_csv_text_source.py` (клас `HFCsvBatchSource`)

**Що робимо:**

- Завантажуємо CSV з Hugging Face Hub (`mechark/controversial_statements`, файл `probing_dataset.csv`).
- Читаємо через pandas з `on_bad_lines="skip"` і `engine="python"`, бо офіційний Dataset Viewer на Hub інколи падає на рядках із зайвими комами в тексті.
- З колонки **`text`** формуємо батчі рядків; з **`label`** (якщо є) передаємо мітки далі по ланцюжку.

**Чому так:**

- Потрібен відтворюваний, легкий корпус для зондування.
- Робастне читання CSV важливіше за строгий viewer на сайті Hub.

---

### Крок B. Активації з Mamay — `MamayActivationProcessor`

**Файл:** `src/pipelines/processors/mamay_activation_processor.py`

**Що робимо:**

- Через `MamayClient` (`src/api/services/mamay_client.py`) робимо HTTP `POST` на `{MODEL_ENDPOINT}/activations` зі списком текстів.
- З відповіді збираємо тензори для шарів `layer_{TARGET_LAYER-1}` (вхід блоку) та `layer_{TARGET_LAYER}` (вихід / резидуал після блоку в нашій номенклатурі API).

**Що йде в SAE:** використовуються **`output_tensors`**, тобто **`layer_{TARGET_LAYER}`**, бо це узгоджується з тренуванням SAE на `blocks.{L}.hook_resid_post` для того ж `L` (див. `configs/saelens_sae_trainer_full.yaml`: `hook_name: blocks.33.hook_resid_post` при `TARGET_LAYER=33`).

**Чому не лише «один шар» у сенсі одного тензора:** API повертає кілька точок; ми явно беремо пару для сумісності з існуючим кодом, але для лейблінгу критичний саме **вихід цільового шару**, який відповідає `hook_resid_post`.

**Прогон через мережу:** повний forward **не** виконується локально на машині з пайплайном — лише запит до сервера Mamay.

Також процесор **прокидає далі** `texts` і `labels`, щоб наступні кроки могли зібрати повний запис для JSONL.

---

### Крок C. Проєкція через SAE — `SaeFeatureEncodeProcessor`

**Файл:** `src/pipelines/processors/sae_feature_encode_processor.py`

**Що робимо:**

1. **Завантаження SAE:** `huggingface_hub.snapshot_download` репозиторію `SAE_HF_REPO_ID` на фіксований коміт `SAE_HF_REVISION`, далі `SAE.load_from_disk(...)` (SAELens). Так ми гарантуємо відтворюваність ваг навіть при плаваючому `main` на Hub.
2. **Пристрій:** `SAE_DEVICE=auto` → CUDA, інакше MPS (Apple Silicon), інакше CPU; при помилках MPS для `encode` модель може бути переведена на CPU.
3. **Інференс:** для кожного тексту беремо тензор активацій форми `[seq_len, d_in]`, додаємо batch-вимір і викликаємо `sae.encode(x)` без градієнтів.
4. **«Вистріли» ознак:** для кожного токена беремо рядок `feature_acts[token] ` і **індекси всіх координат, де значення суворо більше нуля** (`row > 0`). Це перелік цілих індексів у `fired_features` у JSON.

**Токенізатор для підпису токенів:** `transformers.AutoTokenizer` з `NEURON_LABEL_MODEL_NAME` (за замовчуванням `google/gemma-2-9b`), щоб у виході були `token_str` і `token_id`, узгоджені з довжиною послідовності активацій (з евристиками вирівнювання довжин при необхідності). Для Gemma прибирається префікс `▁` (U+2581) у рядковому представленні субтокена для читабельності.

**Поля верхнього рівня запису:**

- `text`, `label`, `target_layer`
- `model` — id моделі токенізатора (як у налаштуваннях)
- `sae_id` — з `NEURON_LABEL_SAE_ID` або похідний ідентифікатор (репо + короткий revision)
- `tokens` — масив об’єктів `token_str`, `token_id`, `fired_features`

---

### Крок D. Запис на диск — `ParquetNeuronActivationSink`

**Файл:** `src/pipelines/sinks/parquet_neuron_sink.py`

**Що робимо:**

- Додаємо файли виду `{stem}_batch_NNNNNN.parquet` у каталог від `NEURON_LABEL_PARQUET_PATH` (див. `settings`).
- Один **рядок** таблиці = один вхідний текст; колонки: `text`, `label` (nullable int64), `target_layer`, `model`, `sae_id`, `tokens` (список структур: `token_str`, `token_id`, `fired_features` — список int).

**Чому Parquet:** компактне зберігання (Snappy), типізовані вкладені структури в Arrow/PyArrow, зручно для аналітики та великих обсягів. Клас `JsonlNeuronActivationSink` залишається в репозиторії для випадків, коли потрібен саме JSONL.

---

## 4. Чому обрано саме такий дизайн

| Рішення | Мотивація |
|--------|-----------|
| Окремий CSV source на HF | Легкий, фіксований датасет для експериментів, без тяжкого стрімінгу |
| Mamay по HTTP | Важка модель на сервері; локально лише SAE + токенізатор |
| `output_tensors` для `TARGET_LAYER` | Узгодження з `hook_resid_post` у тренуванні SAE |
| SAELens `encode` | Офіційний шлях інференсу для завантаженого SAE |
| Поріг `> 0` після `encode` | Стандарт для ReLU/розріджених SAE у практиці SAELens (див. розділ 5) |
| Токенізатор Gemma 2 9B за замовчуванням | Типове наближення до стеку Gemma; можна перевизначити через env |
| Snapshot + commit для SAE | Відтворюваність експерименту |

## 5. Як саме визначається, що «нейрон (ознака) вистрілив»? Який поріг?

**У коді немає окремого настроюваного числового порогу (ні `epsilon`, ні top‑k).**

Критерій зафіксований так (див. `_encode_row` у `sae_feature_encode_processor.py`):

- Після `feature_acts = sae.encode(activations)` для кожного токена береться вектор довжини `d_sae`.
- Індекс `i` потрапляє до `fired_features`, **якщо `feature_acts[token, i] > 0`** (сувора нерівність, нуль не включається).

**Чому це логічно:**

- Для **standard / ReLU-подібних** SAE після `encode` «активні» координати часто інтерпретуються як ті, що мають **додатну** постактивацію; нуль означає «ознака вимкнена» в цьому бінарному сенсі.
- Будь-який додатковий поріг (наприклад `> 1e-6` або top‑p) **наразі не реалізований** — якщо знадобиться, його треба було б додати явно (константа або налаштування в `settings`).

**Що це не робить:**

- Не використовує сирий поріг на **активаціях трансформера до SAE** — рішення приймається **лише в просторі ознак після `encode`**.

---

## 6. Обмеження та ризики інтерпретації

1. **Зміщення токенізатора:** якщо сервер Mamay токенізує інакше, ніж `NEURON_LABEL_MODEL_NAME`, довжини `seq_len` можуть не збігтися — у логах будуть попередження про truncate/pad; `token_id` для «запасних» позицій може бути `-1`, `token_str` — `<pos_k>`.
2. **Кореляція не дорівнює причинності:** активна ознака SAE не «означає концепт» без окремого верифікаційного аналізу.
3. **Розмір рядка:** перелік `fired_features` на токен може бути довгим (типово сотні індексів для розрідженого SAE).
